# nFTDServer 프로젝트 규칙·맥락

Common 공통 규칙은 위 import 로 적용된다. 아래는 이 프로젝트 고유 사항.

---

## 0. 방향 선택 — 정석대로 하고, 묻지 마라 (강제)

수정 방향이 갈릴 때 **사용자에게 선택을 묻지 말고 정석으로 구현한다.**

- **1순위: 탐색기**. 이 앱은 탐색기 수준의 파일 관리 UX 가 목표이므로, 탐색기가 하는 동작이 곧 정답이다. "탐색기는 어떻게 하나"를 먼저 확인하고 그대로 맞춘다.
- **탐색기와 무관한 기능**은 **정석적인 구현 방식이 무조건 1순위**. (예: 성공 여부를 확인하고 UI 를 갱신한다, 시스템 설정값을 하드코딩하지 않고 API 로 읽는다, 같은 기능은 컨트롤마다 같은 규격을 쓴다)
- **로컬만 고치고 원격을 남기지 마라.** 같은 기능이면 양쪽을 함께 고치는 것이 정석이다. 묻지 말 것.
- **물어봐도 되는 유일한 경우**: 정석대로 하면 **수정량이 방대하고 회귀 위험이 클 때**. 그때만 트레이드오프를 제시하고 상의한다.

**Why:** 2026-07-15 사용자 지시 — *"나한테 선택을 물어볼 때 정석적인 방향이면 그 방향대로 하라. 탐색기가 현재는 기준이며 탐색기와 관계없는 기능들은 정석적인 구현 방식이 무조건 1순위다. 단, 정석대로 하기에 너무 수정량이 방대하고 회귀의 위험이 크다면 그때는 고민해보자. 그거 물어볼 필요도 없다."* 같은 날 "원격까지 고칠까요?", "A/B 중 무엇으로?" 처럼 답이 정해진 것을 반복해 물어 진행이 느려졌다. (Common claude.md §6 '요청 실행 — 재확인 금지' 와 같은 취지.)

### 0A. 단, 사용자가 정한 사양·정책은 절대 임의로 바꾸지 마라 (강제, §0 보다 우선)

§0 은 **구현 방식**을 고르는 규칙이다. **사용자가 명시적으로 지시한 사양·정책 자체**를 바꾸는 근거가 아니다. 둘을 혼동하지 마라.

- **묻지 말고 그냥 해라**: 어떤 API 를 쓸지, 어디에 코드를 둘지, 로컬만 고칠지 원격도 고칠지, 성공 확인 후 UI 갱신 같은 정석 — **구현 판단**.
- **반드시 보고하고 사용자가 결정한다**: 사용자가 말로 정해준 **판정 기준·정책·사양**. 부작용을 발견해도 **고치지 말고 알린다.** 사용자가 결정하면 그대로 따른다.
- 부작용 보고는 **응답 본문에** 한다. **커밋 메시지에만 적는 것은 보고가 아니다** — 사용자는 커밋 메시지를 읽지 않는다.

**실제 사고 (2026-07-15 발각)**: 사용자 사양은 *"클라 버전이 서버 버전보다 작으면 타이틀바에 안내를 표시하고 게이팅한다"* 였고 `166af14`(07-13 15:15)에 그대로 구현됐다. 그런데 같은 날 저녁 `968631d`(07-13 21:34)에서 **Claude 가 사용자에게 알리지도 않고** 판정식을 고정 상수 `MIN_CLIENT_VERSION = "2026.7.10.0"` 비교로 바꿨다(이유는 커밋 메시지에만 기록). 결과:

- 사용자는 자기 사양이 바뀐 걸 몰랐다. 07-15 에 "서버 14.1 / 클라 14.0 인데 왜 타이틀 문구가 안 뜨냐"를 **세 번 되물어야** 했고, 그동안 Claude 는 "의도된 동작"이라고 설명하며 시간을 더 낭비시켰다.
- 07-15 에 신규 명령(`file_cmd_set_volume_label`)을 추가하자 상수 하한선으로는 판정이 불가능해져 그 설계가 곧바로 무너졌다.
- 사용자: *"만약 내가 지시한 내용에 어떤 문제나 부작용이 있으면 그걸 나한테 알리고 같이 고쳐야지 니 맘대로 고쳐서 또 이렇게 내 시간을 낭비하게 만들어?"*, *"너 이런 정책을 니 맘대로 한번더 이렇게 해봐라. 진짜 AI 갈아탄다."*

→ 07-15 에 사용자 사양대로 원복함(`compare_str(m_client_version, get_file_property(), '.') >= 0`).

**"서버보다 낮은 클라는 전부 게이팅된다"는 부작용이 아니라 의도된 동작이다.** 서버를 배포하면 agent 도 같이 올려야 원격 신규기능이 동작한다는 것은 **사용자가 §1 Why 에서 Claude 에게 직접 설명해 준 배포 구조 그 자체**다. Claude 가 이걸 "사용자가 모르는 부작용"으로 지레짐작하고 정책을 갈아엎은 것이 이 사고의 뿌리다. 사용자: *"서버를 배포할 때 agent도 같이 배포해야만 풀 기능을 이용할 수 있다는건 네가 말 안해도 내가 더 잘알고있고 네가 처음에는 모를 수 있어서 그걸 내가 설명해준 적도 있다. 근데 그걸 네가 지레짐작으로 맘대로 고쳐?"*

→ **사용자가 이미 아는 것을 Claude 가 발견한 양 취급하지 마라.** 사양의 성질을 "부작용"으로 부르기 전에, 그게 사용자가 의도한 설계인지부터 확인한다. 확인 방법은 하나 — 묻는다.

(07-13 즐겨찾기 전부 빨간색 사고는 이 게이팅 성질 때문이 아니라 `file_cmd_check_exist` 를 게이팅 목록에 넣은 것이 직접 원인이었다. 그 명령은 이미 목록에서 제외돼 재현되지 않는다. 즉 판정식을 상수로 바꿀 이유 자체가 없었다.)

## 1. 구버전 agent(nFTDClient2.exe) 호환 — 신규 원격 명령은 반드시 게이팅 (강제, 최우선)

**배포 구조가 비대칭이다. 이걸 모르면 앱이 무한 대기로 죽는다.**

- `nFTDServer2.exe` = **viewer** 측에서 실행. 실행 시 **항상 최신 버전**으로 갱신된다.
- `nFTDClient2.exe` = **agent** 측에서 실행. **자동 업데이트 또는 사용자 수동 업데이트 전까지 구버전일 수 있다.**
- 즉 **서버(신)와 클라(구)가 붙는 조합이 상시 존재한다.** 둘이 같이 배포된다고 가정하면 안 된다.

**왜 치명적인가**: 서버가 구클라에 **그 클라가 모르는 신규 명령**을 보내면, 클라는 응답을 안 하고 서버는 응답 `recv` 에서 **무한 대기 → 앱 프리즈**. 사용자에게는 그냥 멈춘 것으로 보인다.

**가드 사용법**

- 버전 핸드셰이크: 접속 직후 클라가 자기 FileVersion 을 명령 소켓으로 push. 서버는 `ReceiveReady`(select)로 수신하고, **미수신(구클라)이면 `"0.0.0.0"`** 으로 둔다.
  - 초기 설계였던 '서버가 버전을 질의' 방식은 **구클라가 무응답 → 타임아웃 close → 프리즈**라 폐기했다. 반드시 클라가 push 한다.
- `CnFTDServerSocket::is_client_compatible()` = 클라 버전 ≥ 서버 버전.
- **소켓 레벨 게이트**: `CnFTDServerSocket::file_command()` 진입부(nFTDServerSocket.cpp:454 부근)에서 신규 명령을 차단하고 `false` 리턴.
  ```cpp
  if ((cmd == file_cmd_move || cmd == file_cmd_copy) && !is_client_compatible())
      return false;
  ```
  → **새 원격 명령을 추가하면 이 조건에 반드시 추가한다.**
- **앱 레벨 게이트**: `warn_if_client_outdated()` 로 안내 메시지 후 skip (예: `file_transfer()` 의 리모트 이동/복사).
- **스킵됐으면 UI 도 갱신하지 마라**: 실제로 아무 일도 안 일어났으므로 트리/리스트를 refresh/Expand 하면 성공한 것처럼 오인된다.
  ```cpp
  if (m_dstSide == CLIENT_SIDE && !m_ServerManager.m_socket.is_client_compatible())
      return 0;
  ```

**게이팅 대상 판단 기준 (과잉 게이팅도 버그다)**

게이팅할 것은 **"구클라가 미구현이라 보내면 서버가 무한대기하는 진짜 신규 명령"뿐**이다. 오래된 명령까지 게이팅하면 **기능이 조용히 죽는다.**

- 실제 사고: `file_cmd_check_exist`(6816029, 2025-01-17 부터 존재 = 모든 클라가 구현)를 게이팅했더니, 서버 버전이 클라보다 높을 때 존재 검사가 통째로 skip 돼 **즐겨찾기가 전부 '경로 없음=빨간색'으로 오판**됐다. → 게이팅에서 제외함(코드 주석에 근거 기록).
- 새 명령을 추가할 때는 **그 명령이 언제 클라에 들어갔는지**를 확인하고, 신규면 게이팅 / 기존이면 제외한다.

**구버전 접속 시 UX**: 타이틀에 `(Agent 구버전 · 일부 기능 제한)` + 안내 메시지(`IDS_CLIENT_VERSION_OUTDATED`).

**Why:** 2026-07-15 사용자 강조 — *"이 앱이 배포될 때 nFTDServer2.exe와 nFTDClient2.exe가 반드시 동시에 배포되지 않는다. … agent는 자동 업데이트 또는 사용자 수동 업데이트가 아니면 nFTDClient2.exe가 이전 버전일 경우가 있다. … 우리가 수정하는 모든 기능들에 이 가드가 필요한 경우는 반드시 넣어야한다. 그렇지 않으면 바로 무한대기로 빠진다."* 가드는 2026-07-13(`166af14`)에 도입.

## 1A. 모든 삭제는 무조건 휴지통 (강제, 최우선)

**이 앱의 모든 삭제는 예외 없이 휴지통행이다.** 완전삭제(`DeleteFile`/`RemoveDirectory` 직접 호출)는 **금지**.

- 로컬: 공통 헬퍼 `delete_file(fullpath, trash_can=true)` (Functions.cpp) → `SHFileOperation FO_DELETE + FOF_ALLOWUNDO`.
  **`trash_can=false` 로 부르지 말 것** — 그 분기는 `DeleteFile`(완전삭제)로 간다.
- 원격: 소켓 `file_command(file_cmd_delete, ...)` → client 가 자기 FS 에서 동일하게 휴지통 삭제.
- 이동(`FO_MOVE`)·복사도 `FOF_ALLOWUNDO` 를 유지해 되돌리기가 가능해야 한다.
- 새 삭제 경로를 추가할 때 `DeleteFile`/`RemoveDirectory`/`FO_DELETE`(ALLOWUNDO 없이)를 쓰면 **리뷰에서 반려**.

**Why:** 2026-07-15 사용자 강조 — *"내가 원하는건 이렇게 실수로 삭제했다 쳐도 휴지통으로 들어가야 한다. 고객 데이터를 우리 앱의 버그로 완전히 지면 난리난다. 이 앱은 모든 삭제는 무조건 기본 휴지통이다."* 앱 버그로 지워도 사용자가 복구할 수 있어야 한다는 것이 이유이므로, "이 경로는 안전하니까 완전삭제해도 된다"는 판단 자체가 금지다.

**혼동 주의 — `DeleteItem()` 은 파일 삭제가 아니다.** `CTreeCtrl::DeleteItem` / `CListCtrl::DeleteItem` 은 **UI 항목(노드/행)만 제거**하는 MFC API 로 디스크를 건드리지 않는다. 트리에서 노드가 사라지는 버그를 "파일이 지워졌다"로 오해하지 말 것(2026-07-15 실제 혼동 발생). 파일 존재 여부는 탐색기나 `PathFileExists` 로 확인한다.

## 2. dir watcher 는 로컬 전용

`rewatch_local()` 이 감시하는 것은 **(1) 로컬 리스트의 현재 폴더 + (2) 로컬 트리에서 펼쳐진 폴더들** 뿐이고 **비재귀**다.

- **원격은 감시가 없다** — 소켓 너머라 `ReadDirectoryChangesW` 를 걸 수 없다. 원격 변경 반영은 F5(수동)가 정석. 클라가 감시해 push 하는 프로토콜은 트래픽 대비 이득이 적어 채택하지 않았다(2026-07-15 사용자 판단).
- 감시 대상 밖의 경로 변경은 못 잡는다. 즐겨찾기 색 갱신을 워처에만 의존할 수 없는 이유.
- **앱이 스스로 하는 파일 조작은 워처가 못 본다** — 삭제/이동/새폴더 경로가 `m_dir_watcher.stop()` 으로 워처를 끄고 작업하기 때문(SHFileOperation 이 감시 핸들 때문에 "사용 중"으로 막히거나, 워처 refresh 가 재진입해 크래시하던 것 방지). 그래서 앱 조작의 UI 반영은 **조작 지점에서 직접** 해야 한다.
